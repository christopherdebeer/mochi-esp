# 11 — M11 scope: porting pet state to C

**Status:** scoping, 2026-05-20. Pre-implementation review.
**Predecessor:** [01-bring-up-plan.md](./01-bring-up-plan.md) M11.
**Successor:** [06-scene-contracts.md](./06-scene-contracts.md) (M11.5).

## What M11 is

The bring-up plan says:

> **M11** — Port `shared/decay.ts` + `shared/engagement.ts` +
> `shared/mood.ts` to C. Pet-internal state. Diegetic-interfaces
> doesn't change this directly; it does add the requirement that
> M11 expose a "current pet state" hook the scene-contract
> renderer can read to pick the right pet expression per frame.

This doc walks each of the three modules, sketches the C shape,
and flags the gotchas that aren't visible from "port to C" alone.

## What the three modules actually are

Re-read of `c15r/mochi:shared/{decay,engagement,mood}.ts` against
the canonical types in `shared/types.ts`:

### `decay.ts` — pure stats projection

- `decayStats(stats, statsAt, asleep, ageDays, now) → Stats` —
  per-hour decay with separate awake/asleep rates, clamped to
  `[0, 100]`, capped at 36 hours of elapsed time.
- `ageMultiplier(days) → number` — smooth curve
  `0.7 + 3.3 · exp(-days / 25)`. Replaces the older step-table
  approach; continuous in `days`.
- `computeAge(bornAt, now) → { days, stage }` — discrete stage
  label (`newborn` / `young` / `adolescent` / `adult` / `elder`)
  on top of `days`.

Constants live as module-level `const`. No allocations, no I/O.
Direct port.

### `engagement.ts` — recency signal from event log

- `recentEngagement(events, now) → number` in `[0, 1]`.
- Iterates an event slice (typically ≤12 entries), weights each
  by `WEIGHTS[kind]`, decays exponentially with a 30-min half-life,
  squashes the sum through `1 - exp(-signal / 2)`.
- Exports `ENGAGEMENT_FLOOR = 0.45`.

Also pure. The hidden assumption: there is an event log. On device
today there isn't — `M12` is "event log persistence (LittleFS)".
See "Dependencies & ordering" below.

### `mood.ts` — substrate → Mood → SpriteKey

- `projectMood(pet, events, now) → Mood` — sleeping short-circuit,
  hard floors (HARD_HUNGRY = 15, HARD_TIRED = 12), engagement
  gate, soft floors (SOFT_HUNGRY = 30, SOFT_TIRED = 25, lonely
  threshold scaled by `1 / ageMultiplier`), then the
  playful / content / curious fallback.
- `resolveSprite(pet, mood, ageStage, now)` — transient overrides
  projected mood when alive; otherwise `moodToSprite(mood, ageStage)`.
- `moodToSprite(mood, ageStage)` — the `content → blushing | happy`
  age-fork is the only non-1:1 mapping.
- `moodHint(mood, seed)` — three-string table per mood.

## Proposed C shape

Three files alongside the existing pure-C utilities in
`firmware/main/`:

```
pet_state.h    — types: pet_stats_t, mood_t, sprite_key_t,
                 age_stage_t, age_t, pet_event_t, pet_t
pet_state.c    — internal-only helpers: clamp, age_multiplier

decay.h
decay.c        — decay_stats(), compute_age()

engagement.h
engagement.c   — recent_engagement()

mood.h
mood.c         — project_mood(), resolve_sprite(),
                 mood_to_sprite(), mood_hint()
```

All return-by-value structs; no heap, no statics. Mirrors the shape
of `font8x8.cpp` and the other pure modules already in the tree.
Compiles into `main` component, no new managed component.

Two pieces are not 1:1 with TS:

1. **Strings.** `mood_t` is an `enum`; `mood_hint()` returns a
   `const char *` into a `static const` table. `sprite_key_t` is
   an enum but the renderer needs the string name (matches the
   server's cell endpoint path, e.g. `/devsprite/cell/pet-v1/<name>`);
   we keep a parallel `sprite_key_name[]` table.
2. **Events.** `pet_event_t` carries only `{ kind, at }` for
   M11 — the substrate's `data: Record<string, unknown>` payload
   isn't consumed by any of the three pure modules. M12 will
   need the wider shape; M11 keeps the projection input minimal.

## Tests

Unit tests for these pure functions are unusually high-value
because they have to numerically match the server's mood
projection or the device will render the "wrong" expression for
the pet. The TS side has tests; we mirror them.

Proposed: a `firmware/main/test_pet_state.cpp` built as a host-mode
unit test (compiled with the host compiler, not ESP-IDF), exercising:

- `age_multiplier(0) ≈ 4.0`, `age_multiplier(180) ≈ 0.7` ± 1e-3
- `decay_stats` matches a TS golden fixture for a known
  `(stats, elapsed, asleep, age)` tuple
- `recent_engagement` matches TS at empty / 1-event / 5-talked /
  mixed-kind fixtures
- `project_mood` matches TS for the six branches (asleep, hard
  hungry, hard tired, soft-floor engaged-skip, age-scaled
  lonely, fall-through to playful/content/curious)

Golden fixtures can be a dumped JSON-ish header generated by a
small Deno script in `c15r/mochi` (`scripts/dump-mood-fixtures.ts`).
That keeps the device golden in lock-step with any future tuning
of constants on the server side without manual transcription.

## Dependencies & ordering

M11 as-described doesn't run usefully in isolation. Three
prerequisites are honest to surface:

1. **An event log on device.** `recent_engagement` needs one.
   `M12 — event log persistence` provides it. Two options:
   - **Order M11 before M12** and ship `recent_engagement` with
     an empty event slice — `eng = 0` always, mood projection
     falls into the unengaged branch. Useful only as a numerical-
     equivalence target until M12 lands.
   - **Order M12 before M11.** Then M11 is immediately useful.
     Strongly preferred.
2. **A pet-state snapshot to project from.** `pet_t.stats`,
   `statsAt`, `lastInteractionAt`, `bornAt`, `asleep`, transient.
   The device today has none of these — `/api/state` isn't
   called yet. M13's "sync model" is where the snapshot lands.
   M11 can be built and tested standalone, but it can't be
   *exercised* on device until there is a snapshot to feed it.
3. **A consumer.** The bring-up plan calls out the M11.5
   requirement: scene-contract rendering reads pet state to pick
   the right expression per frame. Until M11.5, the only consumer
   is the existing M8.5 corner-icon hardcoded UI, which doesn't
   need any of this. Concretely: M11 is the substrate for M11.5;
   landing M11 without M11.5 has no user-visible effect.

This argues for treating M11–M11.5–M12–M13 as a coordinated wave
rather than four serial milestones — and very strongly suggests
**M12 (event log) before M11 (port)**, because the alternative is
a port that can't be exercised against its central input.

## Pet-state hook for the scene-contract renderer

`06-scene-contracts.md` M11.5 item 4 reads:

> **Pet-state hook.** Per `01-bring-up-plan.md` M11 note: the
> compositor needs read-access to the current pet mood/decay
> state so the scene-contract renderer can pick the right pet
> expression cell per frame.

The minimum-surface shape:

```c
/* in pet_state.h */
typedef struct {
    pet_t           pet;        /* current snapshot */
    pet_event_t     events[12]; /* recent slice, newest first */
    size_t          n_events;
    int64_t         now_ms;     /* injected for deterministic tests */
} render_ctx_t;

mood_t        project_mood(const render_ctx_t *ctx);
sprite_key_t  resolve_sprite(const render_ctx_t *ctx, age_stage_t age);
```

The compositor calls `resolve_sprite` once per render, gets a
`sprite_key_t`, and turns it into a cell-name string via
`sprite_key_name[key]`. The current corner-icon code in `main.cpp`
keeps working unchanged — it bypasses the hook and asks for an
explicit cell — so M11 lands without breaking M8.5.

## Estimated effort

- Three C files + headers: ~8 hours.
- Golden-fixture generator on the server side: ~2 hours.
- Host-mode unit tests + CMake wiring: ~3 hours.
- Hook wiring into compositor (stub consumer, real wiring with
  M11.5): ~2 hours.

Total: ~15 hours of focused work. Comparable to M8 (SHTC3) in
scope, simpler in surprise budget (all pure functions, no
peripherals).

## What M11 explicitly does NOT do

- **Mutate pet state.** Care actions (`fed`, `played`, …) still
  go to `mochi.val.run/api/mutate`; device doesn't simulate the
  substrate. M13 is where any local-write-then-sync question
  surfaces.
- **Persist pet state.** No NVS, no LittleFS write. The snapshot
  is RAM-only between syncs.
- **Replace the corner-icon UI.** That's M11.5's job.
- **Run consolidation, drift, place selection.** Server-side
  only — the device only consumes the projected mood.
