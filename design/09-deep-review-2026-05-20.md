# Deep review — 2026-05-20

A consolidated cross-cutting review of the firmware, the design docs, and
the build/release pipeline at HEAD of `main` (last commit `89c7677` —
merging the firmware-build CI workflow). The aim is to surface real bugs,
hidden contracts, and pre-release security gaps in one place so the next
few milestones (M9 voice validation, M11.5 scene contracts, first
non-hobbyist deployment) can pick them off deliberately rather than
re-discover them under field conditions.

Findings are grouped by area. Each carries a file:line citation, a
severity (HIGH / MED / LOW), and a concrete fix. Severity is calibrated
against the project's current stage (single-device hobby deployment); a
"HIGH" here is something that matters before sharing a device with
another human, not a CVE-style enterprise blocker.

## 1 — Firmware correctness

### 1.1 Halt loops with no recovery path (HIGH)

`firmware/main/main.cpp:1074-1077` and `:582-585` enter
`while (true) vTaskDelay(pdMS_TO_TICKS(60000))` on a render or pair-init
failure. The touch loop is not running yet at those points, so the
"touch tap to retry" comment on the pair-init halt is wrong — the
device is bricked until power-cycled. The factory-reset watchdog from
`factory_reset::start()` does still run (it spawned on its own task at
boot), so the user can recover via the 10-second PWR+BOOT chord, but
that escape hatch is undocumented at the halt site.

**Fix:** either reboot after a bounded retry/backoff, or render an
explicit "tap to retry" tile and ensure the touch loop is alive before
parking. At minimum the log line and the comment should name the
factory-reset gesture as the recovery path.

### 1.2 OTA SHA-256 is computed and published but never checked (HIGH)

`firmware-release.yml:88-95` writes `sha256` into `latest.json`, and
`ota_update.cpp:103-119` parses only `version` and `url` from the
manifest. The hash is decoration. `esp_https_ota()` validates the TLS
channel but does not verify any digest, and no signature scheme is
enabled (`CONFIG_SECURE_BOOT_V2` is off in `sdkconfig.defaults`).

**Fix (defence-in-depth, cheap):** extract `sha256` from the manifest,
stream the OTA via `esp_https_ota_begin` / `esp_https_ota_perform` /
`esp_https_ota_get_image_len` and hash as you go, then refuse to
`esp_ota_mark_app_valid_cancel_rollback` unless the digest matches.

**Fix (real):** sign release builds and turn on Secure Boot v2 +
anti-rollback before any non-developer device is in the wild. The
eFuse-burn step is one-way, so the policy decision needs to happen
before shipping.

### 1.3 AEC handle vs "enabled" flag can diverge (MED)

`firmware/main/voice/voice_aec.c` exposes `voice_aec_is_enabled()` and
the mute interlock in `voice_peer.c` shorts to "no fallback mute" when
that returns true. If `voice_aec_init` succeeds but the internal
`aec_create` handle is NULL, `voice_aec_process_in_place` skips frames
while `is_enabled()` still reports true — the half-duplex fallback
never engages and the mic loop sees its own echo. Recent commits
(`24a1d40`, `1184684`, `0c60acd`) intentionally retired the half-duplex
hot path; the assumption that AEC is always healthy when "enabled"
should be made explicit.

**Fix:** make `voice_aec_is_enabled()` AND the handle pointer; or
report a tri-state (off / scaffolded / live) and gate the mute
fallback on the scaffolded case.

### 1.4 SPSC ref-ring tail advance is a relaxed read-then-write (LOW-MED)

`voice_aec.c:103-122` reads `tail` with acquire, computes a drop
count, then stores `tail + drop` with release. The ring is SPSC, so
the only other writer to `tail` is `ring_pull`; on a single core the
race is benign, but the ESP32-S3 is dual-core and `voice_peer`'s
producer/consumer can land on different cores. A consumer that drains
between the load and the store will see its progress clobbered.

**Fix:** either use a compare-exchange on `tail`, or move the overrun
policy to "advance head, leave tail alone" (drop newest, not oldest,
which is also a defensible AEC choice and removes the race entirely).

### 1.5 Unbounded leak in pending-tool-args realloc (MED)

`voice_peer.c` pending_append_args reallocs the per-tool args buffer
and silently returns on allocation failure. The old buffer is
preserved (realloc keeps it) but the function reports no error, so
the partially-accumulated tool call continues to grow against a stale
capacity check on the next chunk. In practice the tool call will be
dispatched with truncated JSON and the tool router will drop it; the
session keeps going. Worth tightening, not urgent.

**Fix:** on realloc failure, mark the pending tool as poisoned and
have the dispatcher drop it on `response.done`.

### 1.6 ESP_ERROR_CHECK on recoverable NVS errors (MED)

`nvs_creds.cpp` calls `ESP_ERROR_CHECK(nvs_flash_erase())` and then
`ESP_ERROR_CHECK(err)` on the re-init. Either failure aborts the SoC.
The boot flow can recover from "NVS empty" (provisioning re-runs) far
more gracefully than from a panic; the current behaviour turns a
recoverable layout-change into a hard brick that the user reads as
"my device is dead".

**Fix:** demote to `ESP_LOGW` + `esp_restart()` after a short delay so
the next boot enters provisioning cleanly.

### 1.7 Memory ownership of lifetime allocations is undocumented (LOW)

`main.cpp:277` `new epaper_driver_display(...)` and the PSRAM
framebuffer family at `:642-660` are never freed. Intentional — they
live until the SoC resets — but the pattern reads as a leak. Add a
one-line comment at the allocation sites tagging them as
"device-lifetime, freed by reset" so a future grep for `new` /
`heap_caps_malloc` doesn't spawn cleanup PRs.

## 2 — Security

### 2.1 SoftAP is open and accepts the OpenAI key over plain HTTP (HIGH)

`wifi_prov.cpp:331` sets `authmode = WIFI_AUTH_OPEN` with
`pmf_cfg.required = false`, and the provisioning form (`:215-225`)
collects the WiFi password and OpenAI API key over an unencrypted
HTTP POST on `192.168.4.1`. Anyone within ~10 m during provisioning
can join the AP and either (a) submit forged credentials or (b)
sniff the legitimate user's submission. The DNS hijack on port 53
makes phishing the captive portal trivial from any device on the AP.

**Fix:** generate a per-device WPA2 password at boot, render it on
the e-paper alongside the SSID, and require it for the AP. The
existing splash screen already has the surface to display it.

### 2.2 No CSRF / origin check on the provisioning POST (MED)

`wifi_prov.cpp:234-276` accepts `POST /portal` with no Origin/Referer
check and no token round-trip. With the open AP issue above this is
mostly redundant, but it matters even if 2.1 is fixed — a rogue page
on a phone connected to the SoftAP could auto-submit.

**Fix:** mint a single-use token in the GET handler, store in
`pending_creds`, require it on POST.

### 2.3 OpenAI key + WiFi creds stored unencrypted in NVS (MED)

`openai_key.cpp` and `nvs_creds.cpp` use plain NVS strings.
`CONFIG_NVS_ENCRYPTION` is not set in `sdkconfig.defaults`. Anyone
with momentary physical access (UART/JTAG, or chip-off) can recover
the key, the WiFi password, and the pet pairing token.

**Fix (pre-deployment):** enable NVS encryption + flash encryption
together — NVS encryption alone is pointless without flash
encryption protecting the key partition. This is the same one-way
eFuse decision as Secure Boot in §1.2; bundle them.

### 2.4 `pet_id` logged in plaintext (LOW)

`pair_creds.cpp:27,45` logs the `pet_id` on every load/save. The
OpenAI key path (`openai_key.cpp:26`) already gets this right with
a length+prefix digest. Match that pattern. (The `pet_id` is the
device's bearer credential to `mochi.val.run`; the USB serial log
is the same surface as `dmesg` on a desktop.)

### 2.5 No TLS pinning on any client (MED, but accept-for-now)

`device_pair.cpp:123`, `sprite_fetch.cpp:59`, `ota_update.cpp:82,140`,
and the voice signalling path all use `esp_crt_bundle_attach` with
no pinning. The bundled Mozilla store is the right default for a
val.town-hosted backend whose cert chain is opaque, but it means any
of ~140 CAs can issue a valid cert for `mochi.val.run`.

**Decision needed:** accept as the working tradeoff and document
why (CA bundle is the only sane choice while val.town manages the
cert), or pin the SPKI of the leaf at build time and accept that
val.town cert rotations require a firmware release. For a hobby
device the former is fine; this should be a one-line note in
`00-architecture.md` so it's not relitigated later.

### 2.6 Device auth is a spoofable MAC / pet_id (MED)

`device_pair.cpp:17-18` uses the WiFi MAC as `hw_id`; `voice.cpp`
sends `X-Pet-Id` as the bearer to the voice tool router. Both are
trivially cloneable. Tolerable while the server is single-user, but
the server side will need a real handshake (HMAC over a nonce, or
a per-device cert in the eFuse) before the multi-device sync work
in M13.

### 2.7 Sprite-fetch auto-follows redirects (LOW)

`sprite_fetch.cpp` leaves `disable_auto_redirect = false`. Sprites
are served by val.town and shouldn't 3xx; if they ever do, that's
worth surfacing rather than silently following.

**Fix:** set `disable_auto_redirect = true` for `/devsprite/*`
fetches. Leave OTA alone — GitHub Releases legitimately redirects
to `objects.githubusercontent.com`.

## 3 — Design vs implementation drift

### 3.1 M9 status disagrees between README and 01-bring-up-plan.md (MED)

`README.md:100` lists M9 (audio loopback) as `next`; `01-bring-up-plan.md`
treats M9.a–M9.h as shipped (and `voice/` is fully implemented, with
`main.cpp:453-455` calling `voice::init()` non-fatally). M14 (OTA)
landed after M9 chronologically and is marked done in both. The README
Status table simply hasn't been touched since the voice work landed.

**Fix:** flip M9 to ✅ in the README Status table; add a one-line
note that on-hardware AEC validation (07-voice-architecture.md:307,
:346, :395) is still pending.

### 3.2 Pairing endpoints documented as if they live here (MED)

`README.md:59` ("a `/devsprite/*` family of endpoints… a
`/api/device/pair-*` flow… HTML for `mochi.val.run/pair-device`") and
`04-pairing.md:71-107` describe the server endpoints in tones that
read as "implemented in this repo". `main.tsx` is a 10-line HTML
stub. The real endpoints live in `c15r/mochi`. The cross-reference
in `README.md:138` does name `c15r/mochi`, but the pairing doc
itself does not say "external repo".

**Fix:** prepend a "Server-side: see `c15r/mochi`" note to
`04-pairing.md` and update `README.md:59` to make the split
explicit. Optional: turn `main.tsx` into a redirect to
`mochi.val.run/pair-device` so the stub doesn't look load-bearing.

### 3.3 Dead `06-sync-model.md` reference (LOW)

`00-architecture.md:73` cross-links to `06-sync-model.md`. The slot
is now `06-scene-contracts.md`; the sync model is deferred to M13
without a current doc.

**Fix:** rephrase to "sync model deferred to M13; see Status table".

### 3.4 AEC: compiled-in vs validated (LOW)

`07-voice-architecture.md` says AEC is engaged but on-hardware
validation is pending. Code path is default-on (`VOICE_AEC_USE_ESP_SR=1`).
There's no runtime switch, so a regression in AEC reliability can
only be diagnosed by rebuilding. Add a `voice_aec_set_enabled(bool)`
backdoor wired to a long-press or a build flag so the next bring-up
session can A/B without a reflash.

### 3.5 `.local` claim in 03-provisioning.md (LOW)

`03-provisioning.md:30` mentions `mochi.local`; the same doc later
defers mDNS (`:129`). The opening paragraph reads as if mDNS works
today. Trim the lead to match the deferred reality.

## 4 — Build / CI

### 4.1 CI builds, doesn't analyse (LOW)

`firmware-build.yml` runs `idf.py build` + `idf.py size` and uploads
artefacts. There's no static-analysis pass (clang-tidy, cppcheck,
`-Wextra -Wformat=2 -fstack-protector-strong`) and no `idf.py
build` with `-Wno-error` flipped off, so warnings can accumulate
silently. The codebase is small enough that even a basic clang-tidy
pass would catch most of the §1 findings.

**Fix (cheap):** add `-Wall -Wextra -Wformat=2` to the build step's
CFLAGS and let the warnings fail the build. Defer cppcheck/clang-tidy
until the dust settles.

### 4.2 No anti-rollback eFuse version, only image-state rollback (LOW)

`CONFIG_BOOTLOADER_APP_ROLLBACK_ENABLE=y` rolls back on a *failed*
boot (image stays in PENDING_VERIFY). It does not stop a deliberate
downgrade to a *working* older image (e.g., a compromised release
flipped back to a version with a known bug). The eFuse-burned
`secure_version` field is the right tool, but only matters once
Secure Boot is on (§1.2).

### 4.3 Heap / stack canaries not explicit (LOW)

`sdkconfig.defaults` doesn't set `CONFIG_HEAP_POISONING_LIGHT` or
`CONFIG_STACK_CHECK_STRONG`. ESP-IDF defaults are reasonable but
making them explicit in the file is one line each and means a
future `sdkconfig` regeneration doesn't silently turn them off.

### 4.4 OTA manifest URL is hard-baked (LOW)

`main.cpp:97-98` hardcodes the GitHub manifest URL. If the repo is
renamed or moves orgs, deployed devices stop updating with no
recovery short of USB reflash. Move to NVS with the current URL as
the default, so a provisioning re-run can update it.

## 5 — Prioritised fix list

The picks that pay off before any non-developer ever holds the device:

1. **§1.2** — at minimum, verify the published SHA-256 of the OTA
   image. Real signing + Secure Boot v2 is the durable answer.
2. **§2.1** — close the open provisioning AP. Display a per-device
   WPA2 password on the e-paper at boot.
3. **§1.1** — replace the halt loops with reboot-on-failure or a
   surfaced retry tile.
4. **§3.1**, **§3.2**, **§3.3** — five-minute doc patches. Do them
   while the context is fresh.
5. **§2.4** — match the OpenAI-key log redaction pattern for `pet_id`.
6. **§2.3** — schedule the NVS-encryption + flash-encryption +
   Secure Boot eFuse burn together. One trip, three benefits.
7. **§4.1** — turn on `-Wextra -Wformat=2` in CI now, before the
   warnings pile up.

Everything else is "address as part of the next milestone touching
that area" rather than "fix today".
