# Embedded asset blobs

This directory holds binary files compiled into the firmware via
CMake's `EMBED_FILES`. They are *build artifacts*, generated by
scripts in `firmware/scripts/`, not source you edit by hand.

## `splash.bin` — boot splash framebuffer

200×200, 1-bit packed MSB-first, 5000 bytes. Wire format matches
`/devsprite/splash-v1/boot`: `0` = ink, `1` = paper. Embedded by
`CMakeLists.txt`, consumed by `epd_ui::render_boot_splash()` via the
`_binary_splash_bin_start` linker symbol.

### Generating it

```sh
./firmware/scripts/refresh-splash.sh
```

The script fetches from `https://mochi.val.run/devsprite/splash-v1/boot`
and writes the bytes here. If splash-v1 doesn't have artwork uploaded
yet, falls back to `/devsprite/test` (96×96 fox + caption) so
first-time builds still work.

Re-run the script whenever the `splash-v1` sheet's artwork changes
on mochi. The OTA fetch path (planned) will keep deployed devices
in sync at runtime; this script is only for the bundled default
that ships with the firmware image.

### Why it's not committed

`splash.bin` is regenerated from a known URL. Committing 5 KB of
binary to the val isn't free (no diff visibility, no merge clarity,
no relationship to the source artwork). The script makes the
generation explicit, reproducible, and version-controlled via the
upstream sheet template + uploaded PNG.

### Future: OTA override

When the OTA refresh layer lands, `render_boot_splash()` will
prefer `/lfs/splash.bin` (fetched after WiFi comes up, HEAD-probe
+ ETag invalidation, same pattern as `sprite_cache`) and fall back
to the embedded blob. The embedded blob becomes the offline
fallback; the LittleFS copy carries per-pet variants and seasonal
updates.
